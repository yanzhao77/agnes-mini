需要修正/补充的问题
1. Python 版本要求过高 ⚠️
Python 版本要求: >= 3.13

Python 3.13 于 2024 年 10 月发布，当前（2026年）主流环境可能还未全面跟进。建议降为 >= 3.11 或 >= 3.10，以覆盖更广的用户群体。

2. 视频生成的关键参数描述不一致
位置	说法
5.4 节	num_frames <= 441，需满足 8n+1 规则
5.4 节常用时长	3秒: 81帧，5秒: 121帧，10秒: 241帧，18秒: 441帧
验证：81 = 8×10+1 ✅，121 = 8×15+1 ✅，241 = 8×30+1 ✅，441 = 8×55+1 ✅

建议：显式说明公式为 num_frames = 8n + 1，其中 n 为正整数。

3. 视频查询端点不一致
文档中说查询端点有两条路径：

text
GET /agnesapi?video_id=<VIDEO_ID>
GET /v1/videos/<TASK_ID>
如果实际同时支持两者，建议明确哪个是推荐方式，以及 task_id 和 video_id 的关系

如果不确定，需要实际测试确认哪个端点在当前可用，避免集成时踩坑

4. ImageProvider 的 return_base64 设计
ImageRequest 中有 return_base64 参数

是否真的需要？Base64 数据可能导致内存压力（尤其是 4K 图片）

建议默认 False，仅在需要时开启，并添加警告注释

5. Orchestrator 的意图识别与规划职责重叠
文档描述 Orchestrator 同时做意图识别和任务规划，这两个步骤本质上都调用了同一个 LLM（agnes-2.0-flash）：

如果意图识别 + 任务规划分两次调用 → 增加延迟和 token 消耗

建议考虑合并为一次调用，在 System Prompt 中让模型同时输出"意图类型 + 任务列表"，减少往返开销

6. 缺少环境变量验证逻辑
src/config.py 中，如果 agnes_api_key 为空，系统应该如何表现？

自动 fallback 到 Mock 模式？

还是抛出 ConfigError？

建议在文档中明确这个行为，并在实现中添加相应逻辑。

7. VideoProvider 轮询策略缺少失败处理细节
轮询策略为：2s → 5s → 10s → 30s（上限） → 600s 超时

缺失的场景：

如果任务状态为 failed，是否立即返回错误并携带 error 信息？

如果单次 HTTP 请求失败（网络抖动），是重试还是直接抛出？

建议补充说明这些边界情况。

8. Skill 的 triggers 中英文混用
json
"triggers": ["agnes", "生成图片", "生成视频", "帮我画", "帮我生成视频"]
中文 trigger 可能导致匹配准确性问题（如"帮我画一只猫"能匹配但"画一只猫"不能）。建议：

保持 triggers 作为示例即可

在 SKILL.md 中说明实际匹配逻辑由 Codex 平台控制

9. 缺少 requirements.txt 中的间接依赖锁定
虽然文档列出了主要依赖版本，但建议在 requirements.txt 中锁定传递依赖，或使用 poetry/uv 等工具生成 lock 文件，避免因依赖升级导致兼容性问题。

可选的增强建议
10. 考虑增加资源管理能力
当前 Agent 执行过程中生成的图片/视频 URL 有有效期限制。建议：

增加本地缓存/持久化策略，自动下载并缓存结果

在 AgentResult 中增加 local_path 字段

Provider 层目前没有提到重试策略（尤其是网络抖动场景）。建议：
